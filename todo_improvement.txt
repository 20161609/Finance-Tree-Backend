[Finance Tree 코드 리뷰 정리본]

목적:
이 문서는 Finance Tree 프로젝트의 현재 코드 상태를 기준으로,
실제 터질 가능성이 있는 부분, 구조적으로 개선해야 할 부분,
그리고 취업 포트폴리오 관점에서 우선순위 높게 손봐야 할 부분을 정리한 것이다.

전체 총평:
이 프로젝트는 단순 CRUD 수준을 넘어서,
브랜치 기반 재정 관리, 거래 기록, 영수증 OCR, 리포트 생성까지 포함한 점에서
분명히 강점이 있는 프로젝트다.

특히 다음 부분은 장점이다.
1. 단순 가계부가 아니라 브랜치 트리 구조를 중심으로 한 도메인 설계가 명확하다.
2. 영수증 OCR을 붙여 실제 사용성을 높이려는 시도가 보인다.
3. Excel/PDF 리포트 생성까지 포함되어 있어 결과물을 설명하기 좋다.
4. 작은 조직용 재정관리라는 문제 정의가 비교적 명확하다.

다만 현재 상태는
“기능이 인상적인 데모”에 가깝고,
“안정적인 제품” 단계로 보이기엔 아직 부족한 부분이 있다.

즉,
아이디어와 구현력은 좋지만,
안정성 / 보안 / 데이터 무결성 / 코드 정리도 면에서 보강이 필요하다.


==================================================
1. 가장 먼저 고쳐야 할 부분
==================================================

1-1. report.js 런타임 에러 가능성
문제:
report.js 내부 PDF 생성 코드에서 ENCODING 같은 값이 정의되지 않은 상태로 사용될 가능성이 있다.
또한 import는 되어 있지만 실제로 사용되지 않는 항목들이 여럿 보인다.

영향:
- PDF 생성 시 실제 런타임 에러 발생 가능
- README나 영상에서 “리포트 기능 지원”이라고 말했는데 실제로 깨질 수 있음
- 포트폴리오 신뢰도 하락

조치:
- ENCODING 정의 여부 확인
- 실제 필요 없는 import 전부 제거
- PDF export 흐름을 샘플 데이터로 끝까지 직접 테스트
- 한글 폰트 처리 방식을 하나로 통일


1-2. CORS 설정이 너무 넓게 열려 있음
문제:
허용 origin 목록을 따로 만들어 두고도 실제 middleware에서는 '*' 로 열려 있을 가능성이 있다.

영향:
- 운영 관점에서 보안상 좋지 않음
- 면접에서 보안 고려 질문 시 약점

조치:
- allow_origins를 실제 허용 origin 목록으로 제한
- dev / prod 환경 분리
- 필요 없는 공개 응답 제거


1-3. DB 무결성 제약 부족
문제:
email, branch path 등 핵심 필드에 unique 제약이 부족해 보인다.

영향:
- 중복 이메일 생성 가능
- 동일 사용자 내 중복 branch path 가능
- race condition 상황에서 데이터 꼬일 수 있음

조치:
- Auth.email unique 추가
- Branch에 (uid, path) unique index 추가
- Transaction에 자주 조회하는 조건 기준 index 추가
  예: uid, branch, t_date


1-4. 비밀번호 재설정 방식이 위험함
문제:
임시 비밀번호를 생성해서 이메일로 보내는 방식이면 보안상 좋지 않다.

영향:
- 현대적인 보안 기준에 비해 약함
- 포트폴리오 설명 시 보안 약점으로 잡힘

조치:
- 임시 비밀번호 발송 방식 제거
- reset token 방식으로 변경
- 만료 시간 있는 일회성 토큰 사용
- reset 페이지에서 새 비밀번호 설정


==================================================
2. 논리 버그 / 데이터 오류 가능성
==================================================

2-1. 월별 집계 브랜치 매칭 방식 점검 필요
문제:
branch like 'xxx%' 형태로만 조회하면
예를 들어 Home/A 를 조회할 때 Home/AA 같은 의도치 않은 경로가 섞일 수 있다.

영향:
- 리포트 숫자가 틀릴 수 있음
- 사용자가 구조를 신뢰하기 어려워짐

조치:
- branch == 현재 branch
  또는 branch LIKE '현재branch/%'
  방식으로 명확하게 처리


2-2. 날짜 타입 일관성 부족
문제:
Date 컬럼인데 datetime 객체를 넣는 코드가 있으면 타입 일관성이 깨진다.

영향:
- 지금은 돌아가도 유지보수 시 혼선 발생
- DB별 동작 차이 가능성

조치:
- datetime.strptime(...).date() 로 명확히 date 타입만 넣기


2-3. branch 생성 입력값 검증 부족
문제:
child 값이 빈 문자열, 공백뿐인 문자열, 슬래시 포함 문자열이어도 path 생성이 가능하면 구조가 깨질 수 있다.

영향:
- 잘못된 path 생성
- 이후 트리 구조 꼬임
- 조회 / 삭제 / 이동 로직 이상 가능

조치:
- trim 처리
- 빈 문자열 금지
- '/' 포함 금지
- 예약어 여부 검토


2-4. transaction 수정 시 branch 존재 여부 검증 부족
문제:
거래를 수정하면서 존재하지 않는 branch path로 연결될 수 있다면 데이터 무결성이 깨진다.

영향:
- orphan transaction 발생
- UI에서 조회 안 되거나 이상한 위치에 남음

조치:
- 수정 시 대상 branch 존재 여부 확인
- 없으면 수정 거부


==================================================
3. 프론트엔드에서 아쉬운 부분
==================================================

3-1. login 페이지의 localStorage.clear()
문제:
로그인 페이지 진입 시 localStorage 전체를 clear 하면
앱과 상관없는 저장값까지 전부 날아갈 수 있다.

영향:
- UX 나쁨
- signup 임시 상태값도 날아갈 수 있음
- 다른 기능과 충돌 가능

조치:
- 필요한 key만 removeItem 하도록 수정
- 예: access_token, refresh_token 정도만 제거


3-2. 토큰 저장 방식 개선 여지
문제:
access token을 localStorage에 저장해서 직접 Authorization 헤더에 넣는 방식은
XSS에 취약하다.

영향:
- 보안 관점에서 약점
- 면접에서 지적될 수 있음

조치:
- 당장 전면 교체가 어렵다면 README/면접에서 trade-off 명시
- 향후 HttpOnly cookie + refresh token rotation 구조로 개선 예정이라고 설명 가능


3-3. forget password 흐름 UX 부자연스러움
문제:
비밀번호를 잊은 사용자는 보통 로그인이 안 된 상태인데
해당 요청에 Authorization 토큰을 붙이는 흐름이면 어색하다.

영향:
- 인증 설계가 부자연스럽게 보임
- 실제 사용 흐름과 안 맞음

조치:
- forget password는 비로그인 상태에서도 가능하게 설계
- 이후 reset token 발급 방식으로 정리


3-4. 리포트 숫자/통화 처리 일관성 부족 가능성
문제:
어떤 함수는 숫자를 그대로 쓰고,
어떤 함수는 포맷된 문자열을 넣는 식으로 혼합되어 있을 수 있다.

영향:
- 엑셀에서 숫자 정렬, 합계, 후처리 불편
- 통화 처리 일관성 떨어짐

조치:
- 내부 계산은 숫자 타입 유지
- 출력 직전에만 formatting 적용
- Excel 셀은 가능하면 숫자로 넣고 numFmt 설정


==================================================
4. 구조적으로 개선하면 좋아지는 부분
==================================================

4-1. startup 시 create_all() 직접 실행
문제:
앱 시작 시 Base.metadata.create_all() 방식이면
스키마 변경 이력 관리가 어렵다.

영향:
- 운영 / 협업 / 확장성 약함
- DB migration 경험이 부족해 보일 수 있음

조치:
- Alembic 도입
- schema migration 체계화


4-2. route / crud / lib 역할 경계가 흐림
문제:
비즈니스 로직이 여러 파일에 흩어져 있을 가능성이 있다.

영향:
- 유지보수 어려움
- 중복 코드 증가
- 로직 추적 힘듦

조치:
- route: request/response 처리
- service: 비즈니스 로직
- repository(crud): DB 접근
- storage: firebase / 파일 처리
이렇게 레이어 분리


4-3. branch를 문자열 path 중심으로 저장하는 구조의 한계
문제:
현재는 branch path 문자열 자체를 transaction에 저장하는 방식일 가능성이 높다.

영향:
- branch rename / move 시 연쇄 수정 필요
- 구조 변경 비용 증가

조치:
- 지금 당장 갈아엎을 필요는 없음
- 다만 장기적으로는 branch id FK + path 병행 구조 고려
- README/면접에서는 초기 구현 trade-off로 설명 가능


==================================================
5. 취업 포트폴리오 관점에서의 판단
==================================================

좋은 점:
- 실제 문제를 해결하려는 제품 감각이 있음
- 도메인이 명확함
- 단순 튜토리얼 앱이 아님
- 브랜치 구조, OCR, 리포트까지 묶여 있어 설명 포인트가 많음

아쉬운 점:
- 안정성보다 기능 확장에 조금 더 치우친 인상
- 보안 / 무결성 / 운영성 관점은 보강 필요
- “잘 만든 데모”와 “신뢰 가능한 제품” 사이에서 아직 후자까진 아님

면접관이 볼 가능성:
- “혼자 이 정도 만들었네?” 라는 인상은 줄 수 있음
- 다만 자세히 보면 “운영 감각은 조금 더 다듬어야겠네” 라는 평가도 가능


==================================================
6. 우선순위별 수정 순서
==================================================

[1순위: 당장 터질 가능성 / 신뢰도 문제]
1. report.js 런타임 에러 요소 제거
2. unused import / 죽은 코드 정리
3. 월별 branch 매칭 로직 수정
4. email / branch unique 제약 추가
5. branch 입력 검증 추가

[2순위: 보안 / 데이터 신뢰성]
1. CORS 제한
2. reset token 기반 비밀번호 재설정으로 변경
3. localStorage.clear() 제거
4. transaction 수정 시 branch existence validation 추가

[3순위: 포트폴리오 완성도 상승]
1. Alembic migration 도입
2. service 계층 분리
3. 테스트 코드 3~5개라도 추가
4. README에 문제정의 / 설계 / trade-off / future work 명확히 작성


==================================================
7. 최종 결론
==================================================

이 프로젝트는 충분히 가능성이 있다.
허접한 프로젝트가 아니다.
오히려 방향성과 제품 감각은 꽤 좋다.

하지만 지금 가장 필요한 건
새로운 기능을 더 추가하는 것이 아니라,
이미 있는 기능을 “안정적이고 신뢰 가능하게” 만드는 작업이다.

핵심 한 줄 요약:
아이디어와 구현력은 강점이다.
지금 필요한 것은 완성도, 무결성, 보안, 코드 정리다.

즉,
“더 많이 만들기”보다
“지금 만든 것을 믿을 수 있게 다듬기”
이게 취업 포트폴리오 관점에서 훨씬 중요하다.
